# Research: Dewey Documentation Gaps

**Date**: 2026-04-01

## R1: `uf init` Auto-Detection Behavior

**Source**: PR #63 (`github-unbound-force/unbound-force/pulls/63`), design doc (`openspec/changes/archive/2026-03-28-dewey-auto-sources/design.md`)

**Detection sequence** (runs inside `initSubTools()`):

1. `dewey init` -- creates default single-source `sources.yaml`
2. `generateDeweySources()` -- overwrites with multi-repo config
3. `dewey index` -- indexes all sources

**Detection logic**:

- Scans `../` for directories with `.git/` -- these are sibling repos
- Extracts GitHub org from `git remote get-url origin` (supports SSH and HTTPS)
- Generates per-repo disk sources for each sibling
- Adds org-level disk source (`../`) for org workspace files
- Adds GitHub API source with auto-detected repos from the org
- If GitHub org extraction fails, GitHub source is omitted (graceful degradation)

**Ownership model**: `sources.yaml` is user-owned after creation. `uf init` detects customization by counting `- id:` entries -- if more than 1, assumes user edited it and skips regeneration. Once you customize, `uf init` never touches it again.

**Example generated config** (from PR #63):

```yaml
# Auto-generated by uf init. Customize as needed.
# This file is user-owned -- uf init will not
# overwrite it after initial creation.

sources:
  # Per-repo disk sources (fine-grained provenance)
  - id: disk-local
    type: disk
    name: my-project
    config:
      path: "."

  - id: disk-shared-lib
    type: disk
    name: shared-lib
    config:
      path: "../shared-lib"

  # Org-level files (design papers, plans)
  - id: disk-org
    type: disk
    name: org-workspace
    config:
      path: "../"

  # GitHub API (issues, PRs, READMEs)
  - id: github-org
    type: github
    name: my-org
    config:
      org: my-org
      repos:
        - my-project
        - shared-lib
    refresh_interval: daily
```

**Distinction from `dewey init`**: Running `dewey init` alone creates a bare config with just one disk source pointing to `.`. The multi-repo config is only generated by `uf init` (which runs `dewey init` first, then overwrites).

## R2: Web Source Templates

**Go project template**:

```yaml
# Go standard library reference
- id: web-go-stdlib
  type: web
  name: go-stdlib
  config:
    urls:
      - https://pkg.go.dev/std
    depth: 2
  refresh_interval: weekly

# Framework documentation
- id: web-cobra
  type: web
  name: cobra-cli-framework
  config:
    urls:
      - https://cobra.dev/
    depth: 1
  refresh_interval: weekly

- id: web-gorm
  type: web
  name: gorm-orm
  config:
    urls:
      - https://gorm.io/docs/
    depth: 2
  refresh_interval: weekly
```

**TypeScript/JavaScript project template**:

```yaml
# TypeScript handbook
- id: web-typescript
  type: web
  name: typescript-handbook
  config:
    urls:
      - https://www.typescriptlang.org/docs/handbook/
    depth: 2
  refresh_interval: weekly

# React documentation
- id: web-react
  type: web
  name: react-docs
  config:
    urls:
      - https://react.dev/reference/
    depth: 2
  refresh_interval: weekly

# Next.js documentation
- id: web-nextjs
  type: web
  name: nextjs-docs
  config:
    urls:
      - https://nextjs.org/docs/
    depth: 2
  refresh_interval: weekly
```

**URL selection guidance**:

- Use stable documentation roots, not specific page URLs
- `depth: 1` for single-page docs sites, `depth: 2` for multi-level doc trees
- `refresh_interval: weekly` is a good default -- docs don't change daily
- Check `dewey doctor` if fetch errors appear -- the URL may have changed

## R3: Blog Post Angle

**Title**: "Your AI Agent Doesn't Read the Docs"

**Narrative arc** (following the Gaze blog post pattern):

1. **The Problem**: When you use a framework, you keep the docs open in a browser tab. Your AI agent works from training data that may be months old. The agent generates code using deprecated APIs, old patterns, or hallucinated function signatures.
2. **Why This Happens**: AI models are trained on a snapshot. The Go stdlib added `slices.Concat` in 1.22, but an agent trained before that will use `append` loops. React Server Components changed the rendering model, but agents trained on class components generate legacy code.
3. **The Fix**: Index your toolstack documentation and make it available through semantic search. When an agent asks "how do I sort a slice in Go," the answer comes from the current `pkg.go.dev` docs, not from training data.
4. **How Dewey Does This**: Three source types (disk, GitHub, web). Web sources crawl documentation sites, convert HTML to Markdown, and index with embeddings. Semantic search finds relevant API docs even when the query uses different terminology.
5. **Before/After**: Show a concrete example -- an agent generating code without API docs (stale pattern) vs. with API docs (current pattern).
6. **Getting Started**: Install, add web sources to `sources.yaml`, run `dewey index`.

**Tone**: Technical, honest, problem-focused. Not a product announcement. The problem (stale training data) is universal; the solution (semantic search over current docs) is the insight; Dewey is the implementation.
